# Change Management Evidence Summary (MM3772)

## 1) How GitHub Was Used
GitHub served as the central record for planning, implementing, reviewing, and evidencing work. Commits, branches, and pull requests preserved a chronological audit trail aligned to EE22005 skill evidence requirements.

## 2) How Branches and Pull Requests Were Used
Each scoped task was implemented on a dedicated branch, then proposed through a pull request into `main`. This separation supported controlled review, traceability, and clear documentation of rationale and impact.

## 3) How Issues Were Used for Early Controlled Changes
Early implementation stages were initiated from GitHub Issues (including staged gameplay/VGA tasks), giving explicit scope before coding and linking technical changes to planned evidence outcomes.

## 4) How CHANGELOG.md and EVIDENCE_LOG.md Tracked Later Project Changes
`CHANGELOG.md` captured notable project/evidence updates, while `EVIDENCE_LOG.md` tracked structured evidence items, status, and skill mapping. Together they provided a concise but auditable progression from planned to completed evidence.

## 5) How Codex Was Used as an AI-Assisted Development Tool
Codex was used to draft/update documentation artifacts, maintain consistency across evidence files, and accelerate repository hygiene tasks while preserving human-directed scope control and commit-level traceability.

## 6) Critical Reflection (50-100 words)
Using issue-scoped branches and pull requests worked well because it reduced scope creep and made evidence easier to verify against each change. The combined CHANGELOG and evidence log structure also improved assessment readiness by keeping narrative and artifact tracking synchronized. A key improvement area is earlier automation of evidence capture naming/checklists to reduce manual overhead during final hardware validation. AI assistance was most effective for documentation synthesis, but outcomes still depended on disciplined human review and hardware-grounded verification.
